## P0 · Foundation fixes (pre-MVP)

### Why first

The savings story isn't honest until ingestion is symmetric across
providers. Today (per the audit run on 2026-04-25):

- Gemini contributes **0 observations**, hook never fires.
- Codex telemetry-watcher creates **26 orphan rows**, 19 stuck `in-progress`.
- 99–100% of hook-sourced rows have NULL `started_at`, `model`, `system_prompt`.

If we ship MVP with this broken, the savings pill will silently lie
when used with Gemini/Codex, and we'd have to backfill data later.

|---|---|---|---|
| P0-1 | Forward `started_at` from hook payload (1 line per builder) | `src/history/hook-bridge.js:19,51,82` | S |
| P0-2 | Forward `model` from hook payload when available | `src/history/hook-bridge.js`, ingest route | S |

| P0-5 | Drain stuck `in-progress` rows on watcher expiry (set `ended_at`, `status='completed'`) | both watchers | S |

### Acceptance

P0 is **done** when the test matrix below passes for all three
providers × all five personas. No exceptions, no "we'll fix that
later" exits.

### P0 Test Matrix — synthetic round-trip per provider × persona

The matrix runs **3 providers × 5 personas = 15 test runs**, plus a
final SQL verification pass. Each run is a real launch (not a stub),
through the actual UI/launcher, against a synthetic project, with two
prompt exchanges, and a clean exit. Then we inspect the database.

#### Synthetic projects (one per persona domain)

Created under `~/Projects/_synth/` (gitignored, can be regenerated by
`scripts/seed-synth.sh`). Each is a minimal repo with just enough
domain signal that the persona-filter regexes match cleanly.

| Synth project | Domain | Files (minimum) | Notes |
|---|---|---|---|
| `synth_frontend` | frontend | `ui/App.jsx`, `ui/styles.css`, `package.json` | At least 3 commits with file changes touching `ui/*` |
| `synth_backend` | backend | `src/api/server.js`, `src/db/schema.sql`, `package.json` | Commits matching `src/api/*` and `src/db/*` |
| `synth_debug` | debug | `src/handler.js`, `test/repro.test.js`, plus a `BUG.md` | Commits classified as `bugfix` (test-then-fix pattern) |
| `synth_review` | review | `src/old.js`, `src/refactored.js`, plus a `REFACTOR.md` | Commits classified as `refactor` |
| `synth_devops` | devops | `Dockerfile`, `.github/workflows/ci.yml`, `deploy/` | Commits matching deploy/CI regex |

Seeding: `scripts/seed-synth.sh` creates them with a known commit
history (≥3 commits each, classified observations pre-loaded into the
history DB so persona-filtering has data to draw from).

#### Per-run procedure

For each `(provider, persona)` cell:

1. **Pre-load.** Pick the synth project that matches the persona's
   domain (e.g. `debug` persona → `synth_debug`). Confirm the project
   appears in the picker with the expected history candidates (≥3
   bugfix observations for the debug persona, etc.).
2. **Persona pick.** In the wizard, select the persona. The history
   combo should auto-filter to the matching observations. Skills auto-
   fill from persona defaults.
3. **Launch.** Click Launch. Verify a row is inserted into
   `history_session` *before* the agent spawns
   (`status='launching'`, `started_at` populated, `model` populated,
   `historySessionId` env var set on the spawned process).
4. **Prompt exchanges.** Send 2 short prompts:
   - Prompt 1: a domain-specific question that should trigger one
     tool call (`Read` or `Bash`).
   - Prompt 2: a follow-up that requires the agent to use Prompt 1's
     answer (forces a multi-turn observation chain).
5. **Exit.** Cleanly exit the agent (or close the session in the UI).
6. **Verify.** Run the SQL verification queries below; all must pass.
7. **(Optional) Model swap.** If you want to validate that telemetry
   tracks the model field correctly across changes, swap models
   between personas within the same provider.

#### SQL verification queries

Run after each cell, against `~/.agent-office/agent-office.db`:

```sql
-- V1: Session row exists, status completed, all key fields populated
SELECT history_session_id, project_id, persona_id, provider_id, model,
       started_at, ended_at, status, source
  FROM history_session
 WHERE history_session_id = :id;
-- expect: model NOT NULL, started_at NOT NULL, ended_at NOT NULL,
--         status='completed', source IN ('launcher','provider-hook','telemetry-watcher')

-- V2: At least one observation, with non-null type
SELECT COUNT(*) FROM history_observation WHERE history_session_id = :id;
-- expect: >= 2 (one per prompt exchange, give or take)

SELECT type, COUNT(*) FROM history_observation
 WHERE history_session_id = :id GROUP BY type;
-- expect: type IS NOT NULL for all rows

-- V3: Summary row created
SELECT history_session_id, created_at FROM history_summary
 WHERE history_session_id = :id;
-- expect: 1 row

-- V4: No stuck in-progress
SELECT COUNT(*) FROM history_session
 WHERE provider_id = :provider AND status = 'in-progress'
   AND started_at < datetime('now', '-1 hour');
-- expect: 0

-- V5: Observation count for the persona-domain matches expectation
-- (sanity check the test was actually a real interaction)
SELECT COUNT(*) FROM history_observation o
 WHERE o.history_session_id = :id
   AND (o.files_modified IS NOT NULL OR o.type IN ('bugfix','refactor'));
-- expect: >= 1
```

#### Final verification (after the matrix)

```sql
-- F1: All 15 sessions present, all completed
SELECT provider_id, COUNT(*) FROM history_session
 WHERE project_id LIKE 'synth_%'
   AND started_at > :test_start_iso
 GROUP BY provider_id;
-- expect: claude-code=5, codex=5, gemini-cli=5

-- F2: Per-provider null-rate on critical fields
SELECT provider_id,
       SUM(CASE WHEN model IS NULL THEN 1 ELSE 0 END) AS null_model,
       SUM(CASE WHEN started_at IS NULL THEN 1 ELSE 0 END) AS null_started,
       SUM(CASE WHEN system_prompt IS NULL THEN 1 ELSE 0 END) AS null_prompt,
       COUNT(*) AS total
  FROM history_session
 WHERE project_id LIKE 'synth_%' AND started_at > :test_start_iso
 GROUP BY provider_id;
-- expect: null_model = 0, null_started = 0 for all providers
--         (null_prompt may remain >0 — see open call below)

-- F3: Observations per session
SELECT provider_id, AVG(obs_count) AS avg_obs
  FROM (SELECT s.provider_id, s.history_session_id,
               COUNT(o.observation_id) AS obs_count
          FROM history_session s
          LEFT JOIN history_observation o
            ON o.history_session_id = s.history_session_id
         WHERE s.project_id LIKE 'synth_%' AND s.started_at > :test_start_iso
         GROUP BY s.history_session_id, s.provider_id) t
 GROUP BY provider_id;
-- expect: avg_obs >= 2 for all providers

-- F4: No orphan rows
SELECT provider_id, COUNT(*) FROM history_session
 WHERE project_id LIKE 'synth_%' AND started_at > :test_start_iso
   AND status != 'completed';
-- expect: 0 across all providers
```

If F1–F4 all pass, P0 is done and the foundation is honest.

#### Open call for P0 acceptance

`system_prompt` may remain NULL for hook-sourced rows because the hook
payload doesn't include it (it's only known at launcher time). This is
acceptable for P0 — Phase 1 backfills it via `launcher` pre-creating
the row with system_prompt populated, then hooks upsert. We are
**not** changing hook payloads to carry system_prompt in P0; that's
unnecessary churn. Track as a P1 item, not a P0 blocker.

## P1 · MVP

### Goal

v2 UI is the default. Operator can describe a task → rules-only
frontdesk pre-fills → review wizard → launch into in-browser
terminal → see savings number tick up. **Single provider
(claude-code) wired end-to-end through the new contract**;
codex/gemini still work through legacy paths.

### What's deliberately NOT in MVP

These get cut to keep the slice tight; each lands in a later phase.

- **LLM frontdesk stage** (rules-only is enough for headline routing) → P2
- **Local providers / Ollama bridge** → P3
- **Codex + Gemini through new adapter contract** (they keep using existing hooks; ingestion still works because P0 fixed it) → P3
- **abtop-bridge JSON-RPC** (UI shows what it can from session data alone; live ctx% comes from provider response stream, not abtop) → P4
- **Project activity ticker** (gh-bridge integration) → P5
- **Slide-over drawer's full timeline panel** (rail card detail is enough) → P4
- **Frontdesk learning loop / `frontdesk_decision` few-shots** → P5
- **Sprite ambient strip animations** → P5
- **Outcome prompt UI** (auto-classify only in MVP) → P5
- **Web-tab session sharing** → P5

### Acceptance criteria

1. From a fresh start: `npm run dev` brings up `ao-core` on `:3334`
   and the UI on `:5174` (or next free port). The classic office UI
   is gone or behind a `/legacy` flag.
2. Type a task, click Pre-fill, the wizard combos populate (rules-
   only). Click Launch, claude-code spawns in an xterm pane in-
   browser. Type into the terminal; output renders.
3. Session ends. Savings pill increments. Numbers reconcile against
   `launch_budget` rows in SQL.
4. Run for one day of normal use. Savings pill shows a real,
   defensible number, **outcome-weighted** (rejected sessions don't
   count).
5. Re-run the P0 test matrix for claude-code (5 personas) using the
   new launcher path. All cells pass.

## Cross-cutting policies (apply to all phases)

These don't belong to any single phase but constrain every change.

- **No `if (provider === 'claude-code')` branches outside `src/providers/`.** If you write one, you've broken the adapter contract.
- **Every new persisted field gets a migration.** No shadow schema.
- **Every WS event has a corresponding REST snapshot endpoint.** UI must be able to bootstrap from REST and patch from WS, not require WS for first paint.
- **Privacy mode is honored at the rule layer, not the UI.** Never trust the UI to enforce privacy.
- **Outcome weighting is required for any savings claim.** Rejected sessions never count as savings, anywhere.
- **The classic office UI (`/legacy`) stays available for one full release after P1.** Then it gets ripped out in P2 or P3.
